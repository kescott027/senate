# Senate Security Posture

## Integrity Verification System

The senate bootstrap uses a "fetch and execute" pattern — AI agents download `bootstrap/initiator.md` and follow its instructions. This creates a supply chain risk: if the initiator is tampered with, the agent executes malicious instructions with high privilege.

### Defense layers

1. **Primary: `bootstrap/integrity.json`**
   - SHA256 manifest of all 33 critical files (initiator, laws, skeleton, bootstrapping guides).
   - The initiator's first instruction (Step 0) is to fetch this manifest and verify its own hash before doing anything else.
   - Regenerated by `bootstrap/regenerate-integrity.sh`.
   - Enforced by CI — the `integrity-check` workflow fails if any critical file's hash doesn't match the manifest.

2. **Secondary: Out-of-band gist**
   - A GitHub Gist owned by `kescott027` mirrors the hashes for `initiator.md` and `integrity.json`.
   - This gist is a **separate artifact** — compromising the senate repo does not compromise the gist.
   - The initiator verifies against the gist after the primary check.
   - If the gist disagrees with `integrity.json`, the agent halts — this indicates the repo may be compromised.
   - Gist URL is recorded in `integrity.json` under `secondary_verification.gist_url`.

3. **CI enforcement: `.github/workflows/integrity-check.yml`**
   - Runs on every push and PR that touches critical paths.
   - Verifies all file hashes match `integrity.json`.
   - Auto-regenerates `integrity.json` on push to main if verification fails (safety net for forgotten updates).

4. **Code ownership: `.github/CODEOWNERS`**
   - Changes to `bootstrap/`, `laws/active/`, `CONSTITUTION.md`, `MEMBERSHIP.md`, `sync/manifest.json`, and `project_management_skeleton/` require review from `@kescott027`.

## Recommended Branch Protection Rules

These should be configured in GitHub repo settings for the `main` branch:

### Required

- [ ] **Require pull request reviews before merging** — at least 1 review from a CODEOWNER.
- [ ] **Require status checks to pass** — the `verify-integrity` job must be a required check.
- [ ] **Do not allow bypassing** — even administrators should not bypass required reviews for critical paths.
- [ ] **Restrict push access** — only the repo owner and trusted CI service accounts can push to main.

### Recommended

- [ ] **Require signed commits** — provides cryptographic authorship chain.
- [ ] **Require linear history** — simplifies audit trail (rebase or squash only).
- [ ] **Enable branch protection audit log** — monitor for rule changes.

## Updating the Secondary Verification Gist

When `integrity.json` is regenerated, the script prints the hashes that need to be updated in the gist. **This is a manual step by design** — if it were automated via CI, an attacker who compromises CI could also update the gist, defeating its purpose.

The human updating the gist is the trust anchor.

### Update procedure

1. Run `./bootstrap/regenerate-integrity.sh` — note the printed hashes.
2. Update the gist at the URL in `integrity.json → secondary_verification.gist_url`.
3. The gist content should be:
   ```json
   {
     "last_updated": "<ISO-8601 timestamp>",
     "senate_repo": "kescott027/senate",
     "hashes": {
       "bootstrap/initiator.md": "<sha256>",
       "bootstrap/integrity.json": "<sha256>"
     }
   }
   ```

## Threat Model

| Threat | Mitigation |
|--------|-----------|
| MITM / CDN compromise of raw.githubusercontent.com | Primary hash check catches content mismatch |
| Repo write access compromise | Secondary gist check catches coordinated file + manifest tampering |
| Fork confusion (user fetches from wrong repo) | Initiator URL is hardcoded to `kescott027/senate` |
| CI compromise | Gist update is manual (HITL), not automated |
| Gist compromise | Requires separate credentials; gist revision history provides audit trail |
| Subtle law modification | CI integrity check + CODEOWNER review gates |
